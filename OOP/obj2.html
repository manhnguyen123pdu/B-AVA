12.1 ĐÓNG GÓI (ENCAPSULATION)

📌 Đóng gói là kỹ thuật: Gói gọn dữ liệu và phương thức xử lý vào trong một đơn vị duy nhất (class) và ẩn thông tin

Khái niệm

Đóng gói (Encapsulation) là một trong 4 tính chất cơ bản của OOP, là kỹ thuật:

  • Gói gọn dữ liệu (fields) và phương thức (methods) xử lý dữ liệu đó vào trong một đơn vị duy nhất (class)
  • Ẩn thông tin (information hiding) bằng cách hạn chế truy cập trực tiếp vào dữ liệu từ bên ngoài
  • Kiểm soát truy cập thông qua các phương thức công khai (public methods)

Vì sao cần đóng gói?

1. Bảo vệ dữ liệu

Ngăn chặn việc gán giá trị không hợp lệ từ bên ngoài.

Ví dụ không đóng gói 
class Student {
    public int age;  // Public - ai cũng truy cập được
}

// Sử dụng
Student s = new Student();
s.age = -10;  // Tuổi âm? KHÔNG HỢP LỆ!

2. Tính toàn vẹn dữ liệu

Đảm bảo dữ liệu luôn ở trạng thái hợp lệ.

3. Dễ bảo trì

Khi thay đổi cách lưu trữ dữ liệu, chỉ cần thay đổi bên trong class, không ảnh hưởng code bên ngoài.

4. Linh hoạt trong thay đổi

Có thể thay đổi cách triển khai bên trong mà không làm hỏng code sử dụng class.

Nguyên tắc thiết kế chuẩn

"Ẩn dữ liệu, công khai hành vi"

  • Tất cả thuộc tính (fields) nên là private
  • Cung cấp phương thức public để truy cập (getter) và thay đổi (setter) dữ liệu
  • Kiểm tra tính hợp lệ của dữ liệu trong setter
  • Ẩn chi tiết triển khai, chỉ cho thấy những gì cần thiết

Thử nghiệm Encapsulation

Kết quả:

12.2 ACCESS MODIFIERS

Access Modifiers (các từ khóa xác định phạm vi truy cập) trong Java

1. public

Có thể truy cập từ bất kỳ đâu (cùng class, cùng package, khác package, class con)

Sử dụng khi muốn: method, class, field được truy cập rộng rãi

Ví dụ public 
public class Student {
    public String name;  // Ai cũng truy cập được
    
    public void display() {
        System.out.println(name);
    }
}

2. private

Chỉ truy cập được từ bên trong cùng class

Không truy cập được từ class khác, package khác, hay class con

Dùng cho tất cả fields để đảm bảo đóng gói

Ví dụ private 
public class Student {
    private String name;  // Chỉ Student mới truy cập được
    private int age;
    
    // Chỉ các method trong Student mới truy cập được name, age
    public void setName(String name) {
        this.name = name;
    }
}

3. protected

Truy cập được từ:

  • Cùng class
  • Cùng package
  • Class con (kế thừa) - kể cả ở package khác

Thường dùng cho: các thuộc tính, phương thức muốn class con có thể truy cập

Ví dụ protected 
public class Person {
    protected String id;  // Class con có thể truy cập
}

class Student extends Person {
    void display() {
        System.out.println(id);  // OK - vì Student là class con
    }
}

4. default (không ghi gì)

Truy cập được từ:

  • Cùng class
  • Cùng package

Không truy cập được từ package khác, kể cả class con ở package khác

Còn gọi là package-private

Ví dụ default 
class Student {  // default class - chỉ package này dùng được
    String name;  // default - chỉ package này truy cập được
}

Kiểm tra Access Modifiers

Kết quả:

12.3 PHẠM VI TRUY CẬP (BẢNG SO SÁNH)

Modifier Trong class Cùng package Khác package Class con (kế thừa)
private ✅ Có ❌ Không ❌ Không ❌ Không
default ✅ Có ✅ Có ❌ Không ❌ Không*
protected ✅ Có ✅ Có ❌ Không** ✅ Có
public ✅ Có ✅ Có ✅ Có ✅ Có

Ghi chú:

  • *: Class con trong cùng package thì được, nhưng class con ở package khác thì không
  • **: Class con ở package khác thì được truy cập protected, nhưng class không kế thừa thì không

Ví dụ minh họa

Ví dụ đầy đủ về Access Modifiers 
// File: package1/ClassA.java
package package1;

public class ClassA {
    private String privateField = "private";
    String defaultField = "default";
    protected String protectedField = "protected";
    public String publicField = "public";
}

// File: package1/ClassB.java (cùng package)
package package1;

public class ClassB {
    void test() {
        ClassA a = new ClassA();
        // a.privateField;  // LỖI - private
        a.defaultField;      // OK - cùng package
        a.protectedField;    // OK - cùng package
        a.publicField;       // OK - public
    }
}

// File: package2/ClassC.java (package khác)
package package2;
import package1.ClassA;

public class ClassC {
    void test() {
        ClassA a = new ClassA();
        // a.privateField;    // LỖI
        // a.defaultField;    // LỖI - khác package
        // a.protectedField;  // LỖI - không kế thừa
        a.publicField;       // OK - public
    }
}

// File: package2/ClassD.java (class con ở package khác)
package package2;
import package1.ClassA;

public class ClassD extends ClassA {
    void test() {
        // privateField;      // LỖI
        // defaultField;      // LỖI - khác package
        protectedField;      // OK - class con
        publicField;         // OK - public
    }
}

12.4 GETTER & SETTER

📝 Quy tắc viết getter/setter: Ẩn dữ liệu private, cung cấp phương thức public để truy cập

Quy tắc viết getter/setter

1. Getter

  • Là phương thức public để lấy giá trị của thuộc tính private
  • Tên: get + TênThuộcTính (viết hoa chữ cái đầu)
  • Không có tham số
  • Trả về kiểu dữ liệu của thuộc tính

2. Setter

  • Là phương thức public để gán giá trị cho thuộc tính private
  • Tên: set + TênThuộcTính (viết hoa chữ cái đầu)
  • Có 1 tham số cùng kiểu với thuộc tính
  • Không trả về giá trị (void) hoặc có thể trả về this để chain method

3. Ví dụ cơ bản

Ví dụ cơ bản về Getter/Setter 
public class Person {
    private String name;
    private int age;
    private boolean active;
    
    // Getter cho name
    public String getName() {
        return name;
    }
    
    // Setter cho name
    public void setName(String name) {
        this.name = name;
    }
    
    // Getter cho age
    public int getAge() {
        return age;
    }
    
    // Setter cho age
    public void setAge(int age) {
        this.age = age;
    }
    
    // Getter cho boolean (thường dùng "is" thay vì "get")
    public boolean isActive() {
        return active;
    }
    
    // Setter cho boolean
    public void setActive(boolean active) {
        this.active = active;
    }
}

Kiểm soát dữ liệu đầu vào trong setter

Setter với validation 
public class Student {
    private String name;
    private int age;
    private double gpa;
    
    // Setter với validation
    public void setName(String name) {
        if (name == null || name.trim().isEmpty()) {
            throw new IllegalArgumentException("Tên không được để trống");
        }
        if (name.length() < 2 || name.length() > 50) {
            throw new IllegalArgumentException("Tên phải từ 2-50 ký tự");
        }
        this.name = name;
    }
    
    public void setAge(int age) {
        if (age < 0 || age > 150) {
            throw new IllegalArgumentException("Tuổi phải từ 0-150");
        }
        this.age = age;
    }
    
    public void setGpa(double gpa) {
        if (gpa < 0.0 || gpa > 4.0) {
            throw new IllegalArgumentException("GPA phải từ 0.0-4.0");
        }
        this.gpa = gpa;
    }
    
    // Các getter
    public String getName() { return name; }
    public int getAge() { return age; }
    public double getGpa() { return gpa; }
}

Thử nghiệm Getter/Setter với validation

Kết quả:

LỖI THƯỜNG GẶP KHI VIẾT GETTER/SETTER

1. Viết không đúng quy ước

SAI vs ĐÚNG 
// SAI
public String Name() { return name; }  // Thiếu "get"
public void Name(String n) { name = n; }  // Thiếu "set"

// ĐÚNG
public String getName() { return name; }
public void setName(String name) { this.name = name; }

2. Không kiểm tra dữ liệu trong setter

SAI vs ĐÚNG 
// SAI - Không kiểm tra
public void setAge(int age) {
    this.age = age;  // Có thể gán tuổi âm
}

// ĐÚNG - Có kiểm tra
public void setAge(int age) {
    if (age >= 0) {
        this.age = age;
    }
}

3. Trả về đối tượng có thể thay đổi trực tiếp

SAI vs ĐÚNG 
public class Department {
    private List<String> employees = new ArrayList<>();
    
    // SAI - Trả về list gốc
    public List<String> getEmployees() {
        return employees;  // Người dùng có thể sửa list trực tiếp
    }
    
    // ĐÚNG - Trả về bản sao
    public List<String> getEmployees() {
        return new ArrayList<>(employees);  // An toàn
    }
    
    // HOẶC - Chỉ cung cấp phương thức an toàn
    public void addEmployee(String emp) {
        employees.add(emp);
    }
    
    public void removeEmployee(String emp) {
        employees.remove(emp);
    }
}

4. Tạo setter cho thuộc tính không nên thay đổi

SAI vs ĐÚNG 
public class Account {
    private String accountNumber;  // Số tài khoản KHÔNG thay đổi
    
    // SAI - Không nên có setter
    public void setAccountNumber(String num) {
        this.accountNumber = num;
    }
    
    // ĐÚNG - Chỉ có getter
    public String getAccountNumber() {
        return accountNumber;
    }
    
    // Constructor để khởi tạo
    public Account(String accountNumber) {
        this.accountNumber = accountNumber;
    }
}

// Tạo class Product với các thuộc tính:

Kiểm tra bài tập

Giải pháp mẫu:

Tóm tắt kiến thức quan trọng

  • Luôn đặt thuộc tính là private
  • Sử dụng public getter/setter để truy cập
  • Kiểm tra dữ liệu trong setter để đảm bảo tính hợp lệ
  • Không tạo setter cho thuộc tính không nên thay đổi
  • Trả về bản sao cho các đối tượng có thể thay đổi (List, Array, etc.)
  • Tuân thủ quy ước đặt tên: getXxx(), setXxx(), isXxx() (cho boolean)
